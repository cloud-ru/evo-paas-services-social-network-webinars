# Микросервисы Социальной Сети

Это демонстрационное приложение социальной сети для серии вебинаров.

Приложение социальной сети на основе микросервисов, созданное с использованием NestJS, TypeScript и PostgreSQL.

## Обзор Архитектуры

Этот монорепозиторий содержит 7 микросервисов, которые взаимодействуют через TCP:

| Сервис          | Порт | Ответственность                                                 |
| --------------- | ---- | --------------------------------------------------------------- |
| API Gateway     | 3000 | Маршрутизация запросов, валидация аутентификации, публичный API |
| Auth Service    | 3001 | OAuth 2.0, JWT токены, верификация email                        |
| Email Service   | 3002 | Отправка email, шаблоны                                         |
| User Service    | 3003 | Профили пользователей, аватары, статус онлайн                   |
| Message Service | 3004 | Личные сообщения, поиск, лайки                                  |
| Post Service    | 3005 | Посты, файлы, лента                                             |
| File Service    | 3006 | Интеграция с S3, presigned URLs                                 |

## Технологический стек

- **Фреймворк**: NestJS
- **Язык**: TypeScript
- **База данных**: PostgreSQL (с отдельными схемами для каждого сервиса)
- **ORM**: Prisma
- **Взаимодействие**: NestJS Microservices (TCP)
- **Документация API**: Swagger
- **Хранение файлов**: S3 (MinIO для локальной разработки)
- **Пакетный менеджер**: Yarn

## Начало работы

### 1. Установка зависимостей

```bash
yarn install
```

### 2. Настройка переменных окружения

Скопируйте пример файла окружения и настройте его:

```bash
cp env.example .env
```

Настройте следующие переменные в `.env`:

- Строки подключения к базе данных
- Секреты JWT
- Учетные данные S3/MinIO
- Конфигурация email сервиса

### 3. Запуск приложения (Docker - Рекомендуется)

Самый простой способ запустить систему — использовать Docker Compose. Это соберет все образы, сгенерирует клиенты Prisma внутри контейнеров и запустит сервисы.

```bash
yarn start
# ИЛИ
docker-compose up --build
```

### 4. Миграции Базы Данных

**Автоматическое применение**:
Сервисы настроены на автоматическое применение ожидающих миграций при запуске (например, в `prisma.service.ts`).

**Создание новой миграции**:
Чтобы создать миграцию, используйте скрипты для конкретных сервисов. Эти скрипты запускают временный контейнер базы данных для безопасной генерации файла миграции без необходимости запуска локального экземпляра Postgres на определенных портах.

```bash
yarn create-migration:auth
yarn create-migration:user
yarn create-migration:message
yarn create-migration:post
```

### 5. Ручная разработка (Запуск без Docker)

Если вы предпочитаете запускать сервисы отдельно для разработки:

1. **Запуск инфраструктуры**:

   ```bash
   docker-compose up -d postgres minio
   ```

2. **Генерация клиентов Prisma** (Требуется для типизации):
   Вы должны запускать эти команды каждый раз, когда меняете файлы `schema.prisma`.

   ```bash
   yarn prisma:generate:auth
   yarn prisma:generate:user
   yarn prisma:generate:message
   yarn prisma:generate:post
   ```

3. **Запуск сервиса**:
   ```bash
   npx nest start <service-name> --watch
   # Пример:
   npx nest start api-gateway --watch
   ```

## Деплой Docker

### Предварительный этап

Замените `registry_name` на имя вашего репозитория в Artifact Registry.

### Архитектура

Проект использует архитектуру **многоэтапной сборки Docker** с пользовательскими базовыми образами, хранящимися в частном реестре (`registry_name.cr.cloud.ru`):

1. **Базовые Образы** (хранятся в `apps/base-images/`):
   - `base-build`: Node.js 22 Alpine образ для сборки приложений
   - `base-run`: Node.js 22 Alpine образ для запуска приложений

2. **Образы Сервисов**: Каждый микросервис имеет свой собственный Dockerfile в `apps/<service-name>/Dockerfile`, который:
   - Использует `base-build` для этапа сборки (установка зависимостей, генерация Prisma, компиляция TypeScript)
   - Использует `base-run` для этапа выполнения (только dependencies для продакшена)
   - Включает клиент Prisma и файлы миграций для доступа к базе данных

### Команды Деплоя

#### 1. Деплой Базовых Образов (Требуется Сначала)

Перед деплоем любых микросервисов вы должны собрать и отправить базовые образы в реестр:

```bash
# Деплой обоих базовых образов (build + run)
yarn deploy:base

# Или деплой по отдельности:
yarn docker:deploy:base:build   # Сборка, тегирование и отправка образа base-build
yarn docker:deploy:base:run     # Сборка, тегирование и отправка образа base-run
```

#### 2. Деплой Отдельных Микросервисов

Каждый микросервис может быть развернут независимо:

```bash
# API Gateway
yarn deploy:api-gateway

# Auth Service
yarn deploy:auth

# Email Service
yarn deploy:email

# User Service
yarn deploy:user

# Message Service
yarn deploy:message

# Post Service
yarn deploy:post

# File Service
yarn deploy:file
```

Каждая команда деплоя выполняет три шага:

1. **Build**: Создает Docker образ из Dockerfile
2. **Tag**: Тегирует образ для частного реестра
3. **Push**: Отправляет образ в `registry_name.cr.cloud.ru`

#### 3. Деплой Всех Сервисов

Чтобы задеплоить все микросервисы сразу:

```bash
yarn deploy:all
```

Это последовательно развернет все 7 микросервисов (api-gateway, auth, email, user, message, post, file).

### Рабочий процесс Деплоя

1. **Первоначальная Настройка**:

   ```bash
   # 1. Сборка и отправка базовых образов
   yarn deploy:base

   # 2. Деплой всех микросервисов
   yarn deploy:all
   ```

2. **Обновление Базовых Образов** (когда меняется версия Node.js):

   ```bash
   yarn deploy:base
   # Затем пересборка всех сервисов, которые от них зависят
   yarn deploy:all
   ```

3. **Обновление Отдельного Сервиса**:
   ```bash
   # Пересборка только измененного сервиса
   yarn deploy:auth  # или любой другой сервис
   ```

### Локальная Разработка с Docker

Для локальной разработки используйте Docker Compose, который оркестрирует весь стек:

```bash
# Запуск всех сервисов
yarn start
# ИЛИ
docker-compose up --build

# Остановка всех сервисов
yarn down
```

Файл `docker-compose.yml` обрабатывает:

- Сборку всех микросервисов
- Настройку базы данных PostgreSQL
- Настройку MinIO (S3-совместимое хранилище)
- Конфигурацию сети между сервисами
- Переменные окружения

## Принципы Проектирования

- **Domain-Driven Design**: Каждый микросервис представляет одну сущность домена.
- **Разделение ответственности**: У каждого сервиса своя схема базы данных.
- **Паттерн Репозиторий**: Доступ к базе данных абстрагирован через репозитории.
- **Контрактное Взаимодействие**: Общие типы в `libs/types` определяют контракты сервисов.
- **Health Checks**: Каждый сервис предоставляет ендпоинт `/health`.
- **Read/Write Splitting**: Автоматическое разделение запросов на чтение и запись для масштабирования базы данных.

## Масштабирование и Репликация Чтения

Для обеспечения высокой доступности и горизонтального масштабирования, микросервисы (Auth, User, Message, Post) поддерживают работу с репликами чтения базы данных PostgreSQL.

### Как это работает

Мы используем расширение `@prisma/extension-read-replicas`, которое позволяет Prisma автоматически перенаправлять:
- Запросы на запись (`create`, `update`, `delete`) — на **Primary** узел.
- Запросы на чтение (`findMany`, `findUnique`, `count` и др.) — на доступные **Replica** узлы.

### Конфигурация

Конфигурация осуществляется через переменные окружения в файле `.env`:

1.  `*_DATABASE_URL`: URL основной базы данных (Primary). Используется для всех операций записи и миграций.
2.  `*_DATABASE_REPLICA_URLS`: Список URL реплик чтения, разделенных запятой.

**Пример конфигурации для Post Service:**

```env
POST_DATABASE_URL="postgresql://user:pass@primary-host:5432/db"
POST_DATABASE_REPLICA_URLS="postgresql://user:pass@replica-1:5432/db,postgresql://user:pass@replica-2:5432/db"
```

Если `*_DATABASE_REPLICA_URLS` не задан или пуст, сервис будет использовать `*_DATABASE_URL` как для чтения, так и для записи.

### Преимущества

- **Снижение нагрузки**: Тяжелые запросы на чтение (например, получение ленты новостей) распределяются между несколькими узлами.
- **Отказоустойчивость**: Если одна из реплик недоступна, Prisma продолжит работу с оставшимися.
- **Гибкость**: Вы можете добавлять новые реплики на лету, просто обновляя список URL в переменных окружения.

## Добавление Нового Микросервиса

1. Сгенерируйте новое приложение NestJS: `nest generate app <service-name>`
2. Настройте TCP микросервис в `main.ts`.
3. Добавьте схему базы данных в `prisma/schema.prisma` (если stateful).
4. Создайте слой репозитория для доступа к базе данных.
5. Зарегистрируйте сервис в API Gateway.
6. Добавьте контракты в `libs/types`.
7. Создайте Dockerfile для деплоя.
8. Добавьте скрипты в `package.json` для генерации prisma и миграций.
